1. Visão geral da solução
A versão anterior do wireframe estava correta visualmente. A alteração principal agora é que a
contribuição mensal deixa de ser apenas um lançamento manual e passa a ser uma cobrança Pix
individual para cada casa e mês.
A confirmação do pagamento deve acontecer de forma automática por webhook do gateway. O
administrador ainda pode lançar manualmente uma exceção, mas o fluxo principal será: gerar cobrança,
exibir QR Code/Copia e Cola, receber webhook, atualizar status e recalcular saldo.
Etapa
Descrição
1. Admin gera cobranças
Sistema cria 10 cobranças Pix, uma para cada casa, usando o valor mensal
configurado.
2. Morador visualiza Pix
No app aparece a contribuição do mês com QR Code, Pix Copia e Cola,
vencimento e status.
3. Morador paga
Pagamento é feito pelo banco do morador, fora do app.
4. Gateway notifica
Asaas envia webhook para o backend avisando PAYMENT_RECEIVED ou evento
equivalente.
5. App atualiza
Backend marca a cobrança como paga e o app passa a exibir status Pago.
6. Portal dá
transparência
Todos veem valores arrecadados, pendências, despesas, documentos e saldo
calculado.
Base usada para o desenho: a documentação do Asaas informa suporte a recebimento Pix por QR Code dinâmico/estático,
criação de cobrança com billingType PIX, recuperação de QR Code Pix por cobrança e webhooks para eventos como
PAYMENT_CREATED, PAYMENT_RECEIVED e PAYMENT_OVERDUE. A documentação da Efí também informa Pix Cob/CobV,
sandbox e conciliação como alternativa futura.
2. Decisão técnica recomendada
Para o Codex, peça para criar dois projetos: app mobile e backend. O React Native não deve falar direto
com o gateway, porque isso exporia token/API key e deixaria regras financeiras no celular. O app chama
o backend, e o backend chama o gateway.

5.3 Webhook de pagamento
●
Gateway chama POST /api/webhooks/asaas.
●
Backend valida o token/cabeçalho configurado.
●
Backend registra o evento em pix_webhook_events.
●
Se o evento já foi processado, ignora com sucesso para garantir idempotência.
●
Se o evento for PAYMENT_RECEIVED, marca a cobrança como PAID.
●
Atualiza contribution.status para PAID.
●
Recalcula os totais exibidos no app.
6. Regras de negócio
Regra
Descrição
Cobrança única
Cada casa só pode ter uma cobrança Pix ativa por mês. Usar índice único
house_id + reference_month.
Valor padrão
Usar o valor mensal configurado, permitindo ajuste individual antes de gerar
cobranças.
Status financeiro
A contribuição só entra no saldo quando a cobrança estiver PAID/RECEIVED.
Pendente
Cobranças criadas e não pagas ficam PENDING.
Vencida
Se passar do vencimento ou vier evento PAYMENT_OVERDUE, status OVERDUE.
Cancelamento
Admin pode cancelar cobrança, mas deve registrar motivo.
Pagamento manual
Admin pode marcar como pago manualmente em exceção, mas o sistema deve
registrar manualPayment=true e motivo.
Estorno
Se houver evento de estorno, status REFUNDED e contribuição deixa de compor
saldo.
Idempotência
Todo evento de webhook deve ser processado uma única vez por event_id.
Transparência
Moradores veem valores e status, mas não veem dados sensíveis de tokens, CPF
completo ou chaves de API.

10. Integração com Asaas - fluxo técnico
O backend deve usar variáveis de ambiente para credenciais e URLs. Nunca colocar API key no app
React Native.
ASAAS_ENV=sandbox
ASAAS_BASE_URL=https://api-sandbox.asaas.com/v3
ASAAS_API_KEY=colocar_no_secret_manager_ou_env
ASAAS_WEBHOOK_TOKEN=token_secreto_configurado_no_webhook
APP_PUBLIC_API_URL=https://api.sua-vila.com.br
10.1 Criar cobrança Pix
Para cada casa/mês, o backend deve criar uma cobrança com billingType PIX, valor, vencimento e
externalReference. A externalReference deve ser previsível para conciliação, por exemplo:
VILA-2026-05-HOUSE-01.
POST /v3/payments
{
  "customer": "cus_xxx",
  "billingType": "PIX",
  "value": 100.00,
  "dueDate": "2026-05-10",
  "description": "Caixinha da Vila - Casa 01 - Maio/2026",
  "externalReference": "VILA-2026-05-HOUSE-01"
}

11. Segurança e auditoria
●
API key do gateway apenas no backend, nunca no app.
●
Webhook com validação do header asaas-access-token ou mecanismo equivalente definido no painel do
gateway.
●
Registrar todos os payloads de webhook em webhook_events para auditoria.
●
Implementar idempotência por event_id e também por gateway_payment_id + eventType.
●
Não armazenar CPF completo no app. Se necessário, armazenar apenas mascarado.
●
Permitir que moradores vejam a transparência financeira, mas não dados sensíveis de outros
moradores.
●
Usar roles: ADMIN e RESIDENT.
●
Gerar logs de ações administrativas: quem marcou manualmente, quem cancelou, quem alterou valor.
●
Usar HTTPS obrigatório em produção.

Fluxo de geração de cobrança Pix:
1. Admin acessa Cobranças Pix.
2. Seleciona o mês, exemplo 2026-05.
3. Clica em "Gerar cobranças do mês".
4. Backend verifica as 10 casas ativas.
5. Backend cria uma contribuição para cada casa/mês se ainda não existir.
6. Backend cria ou atualiza customer no Asaas para o morador responsável.
7. Backend cria cobrança no Asaas com billingType PIX.
8. Backend salva gatewayPaymentId, externalReference e status.
9. Backend chama /v3/payments/{id}/pixQrCode.
10. Backend salva qrCodeBase64, pixCopyPaste e expirationDate.
11. O morador passa a ver QR Code e Pix Copia e Cola no app.

Fluxo funcional que o Codex deve implementar
O objetivo é evitar que o Codex crie apenas listas genéricas. O app precisa ter um fluxo administrativo
completo para administrar melhorias da vila, desde a ideia inicial até a prestação de contas final.
1. Serviço
planejado
2. Orçamentos
cotações
3. Aprovação
ADMIN
4. Execução
em andamento
5. Despesa
saldo
Passo a passo
ADMIN cadastra uma manutenção ou melhoria planejada, por exemplo: troca do cabo de aço do portão,
pintura da vila, avaliação estrutural, interfone ou toldo.
Dentro do serviço, o ADMIN cadastra um ou mais orçamentos, com fornecedor, valor, validade, previsão
e documento/anexo.
O ADMIN aprova um orçamento. O orçamento aprovado fica vinculado ao serviço como
approvedBudgetId.
O serviço pode mudar para APROVADO, EM_ANDAMENTO, FINALIZADO ou CANCELADO, conforme a
evolução real.
Ao finalizar, o ADMIN informa valor final, data de conclusão, fornecedor executado, nota fiscal/recibo e
decide se deseja gerar uma despesa automaticamente.
Quando uma despesa é gerada, ela passa a reduzir o saldo do dashboard e fica vinculada ao serviço, ao
orçamento e ao documento.
Relação com a API Asaas
A API Asaas continua responsável pelas cobranças Pix das contribuições mensais. Orçamentos e serviços não
criam cobrança Pix automaticamente. Eles apenas consomem o saldo da caixinha quando viram despesa. O
saldo exibido deve combinar: entradas confirmadas pelo Asaas via webhook menos despesas cadastradas ou
geradas ao finalizar serviços.
